Data Quality Flags

Understanding data quality indicators, allocation flags, and imputation methods in PLAP datasets

Overview

PLAP provides comprehensive data quality indicators to help researchers understand the reliability of each observation and make informed decisions about data usage. Quality flags indicate whether values were directly observed, imputed, allocated, or derived from other sources.

Understanding Quality Flags

For most variables, PLAP provides a corresponding quality flag variable with the suffix _QF. These flags indicate the source and reliability of each value.

Naming Convention: If the variable is TRANSACTION_VALUE, its quality flag is TRANSACTION_VALUE_QF.

Standard Quality Flag Codes

Code Label Description Reliability
0 Not applicable Value is not applicable for this record (e.g., agricultural land area for urban property) N/A
1 Directly reported Value was recorded directly from source document/system High
2 Logically imputed Value derived from other fields using business rules (e.g., area from khasra dimensions) High
3 Allocated from related record Value copied from a related record (e.g., same property, different transaction) Medium
4 Hot-deck imputation Value imputed from a similar record (donor) in the dataset Medium
5 Model-based imputation Value predicted using regression or machine learning model Medium
6 Mean/median substitution Value replaced with district/tehsil average Low
7 Edited/Corrected Original value was edited due to detected error or inconsistency Medium
9 Missing Value is missing and could not be imputed N/A

Allocation Flag Variables

Allocation flags (suffix _AF) indicate when a value has been distributed or shared across multiple records. This commonly occurs when a single transaction involves multiple properties or owners.

Multi-Property Transactions

When a single deed covers multiple properties, the transaction value is allocated proportionally.

Example:

Deed covers 3 plots sold for PKR 30M total

  • • Plot A (100 kanals): PKR 15M allocated
  • • Plot B (60 kanals): PKR 9M allocated
  • • Plot C (40 kanals): PKR 6M allocated

Each record has TRANSACTION_VALUE_AF = 1

Multi-Owner Transactions

When multiple owners are involved, ownership shares and values are allocated per owner.

Example:

Property inherited by 4 siblings

  • • Sibling A: 35% share (male)
  • • Sibling B: 35% share (male)
  • • Sibling C: 15% share (female)
  • • Sibling D: 15% share (female)

Each record has OWNER_SHARE_AF = 1

Allocation Flag Codes

Code Label Description
0 Not allocated Value applies directly to this record without distribution
1 Area-proportional allocation Value allocated based on land/property area ratios
2 Share-proportional allocation Value allocated based on ownership share percentages
3 Equal allocation Value divided equally when proportions are unknown
4 DC-rate weighted allocation Value allocated using DC valuation ratios

Imputation Methodology

Missing values in PLAP data are imputed using a hierarchical approach that prioritizes accuracy and consistency. The imputation method used for each value is recorded in the quality flag variable.

Imputation Hierarchy

1

Logical Derivation (QF=2)

Values that can be calculated from other fields using business rules.

Examples: Total area = length × width; CVT = 3% of transaction value; Age = current year - birth year

2

Related Record (QF=3)

Values copied from a linked record (same property, same owner, or same transaction).

Examples: Property area from previous transaction of same khasra; Owner details from NADRA verification in another deed

3

Hot-Deck Imputation (QF=4)

Value borrowed from a "donor" record with similar characteristics.

Matching criteria: Same tehsil, same property type, similar area range, same year. Donor selected randomly from eligible pool.

4

Model-Based Imputation (QF=5)

Value predicted using statistical models trained on complete cases.

Models used: OLS regression for continuous values (transaction value); Logistic regression for binary values; Random forest for complex patterns

5

Mean/Median Substitution (QF=6)

Last resort: group average substituted when other methods fail.

Note: Use with caution. This method reduces variance and may bias analyses. Consider excluding these cases in sensitivity analysis.

Best Practices for Using Quality Flags

Quality flags enable you to make informed decisions about data usage and conduct sensitivity analyses.

Recommended Practices

  • Include quality flags in your extract for key analysis variables
  • Report imputation rates in methodology section of publications
  • Conduct sensitivity analysis excluding low-quality records
  • Use quality flags as control variables in regression models
  • Check for patterns in missingness (MCAR, MAR, MNAR)

Practices to Avoid

  • Ignoring quality flags entirely in your analysis
  • Treating all imputed values as equivalent to reported values
  • Using mean-imputed values (QF=6) for variance estimation
  • Deleting all imputed cases (loses information unnecessarily)
  • Assuming missing data is random without testing

Example: Filtering by Quality

# R example: Analyze only high-quality transaction values
library(dplyr)

plap_data %>%
  # Keep only directly reported or logically imputed values
  filter(TRANSACTION_VALUE_QF %in% c(1, 2)) %>%
  group_by(DISTRICT) %>%
  summarize(
    n = n(),
    mean_value = weighted.mean(TRANSACTION_VALUE, TXNWT),
    pct_reported = mean(TRANSACTION_VALUE_QF == 1) * 100
  )

# Sensitivity analysis: Compare results with/without imputed values
full_analysis <- plap_data %>%
  summarize(mean = weighted.mean(TRANSACTION_VALUE, TXNWT))

strict_analysis <- plap_data %>%
  filter(TRANSACTION_VALUE_QF == 1) %>%
  summarize(mean = weighted.mean(TRANSACTION_VALUE, TXNWT))

cat("Full sample mean:", full_analysis$mean, "\n")
cat("Reported-only mean:", strict_analysis$mean, "\n")

Sensitivity Analysis Recommendation

For important findings, we recommend running analyses under three scenarios:

  • Full sample: All records including imputed values
  • High quality: Only records with QF = 1 or 2
  • Reported only: Only records with QF = 1

If conclusions differ substantially across scenarios, discuss this in your methodology.
